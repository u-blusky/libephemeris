## Bug Fix 1: Lunar Occultation Frame Mismatch

**File:** `libephemeris/eclipse.py` — `_get_target_position()`, `_batch_separations()`

**Symptoms:** `lun_occult_when_glob()` failed to find the Regulus occultation on
2017-01-19. The function computed a minimum separation too large to qualify as
an occultation.

**Root cause:** Moon position was computed in the equinox-of-date frame
(`epoch="date"`), but star position was computed in J2000 with only proper motion
applied — no precession. By 2017, precession introduces ~0.8° offset in RA
between the two frames. The angular separation calculation mixed frames, making
the Moon appear ~0.8° further from Regulus than it actually was.

This affected both code paths:
- **Scalar path** (`_get_target_position()`, ~line 5737): Used manual RA/Dec
  with proper motion but no frame rotation
- **Vectorized path** (`_batch_separations()`, ~line 6129): Same issue in the
  batch coarse-scan

**Fix:** Both paths now use Skyfield's `Star` class with
`.apparent().radec(epoch="date")`, which applies proper motion, precession,
nutation, aberration, and deflection — matching the Moon's frame exactly.

Additionally, a **boundary guard** was added after the golden-section refinement
step. The ±0.5 day refinement window could drag `jd_max` before `jd_start`
(e.g., finding a May 31 event when searching from June 1). The guard rejects
events that fall outside the search direction.

## Bug Fix 2: Observer Cache Identity Collision

**File:** `libephemeris/cache.py` — `get_cached_observer_at()`

**Symptoms:** Two tests passed individually but failed in the full suite:
- Mars geocentric returned NaN coordinates
- Moon topocentric returned wrong parallax-corrected position

Both failures were non-deterministic and depended on test execution order.

**Root cause:** The observer-at-time cache used `id(observer)` as cache key.
Python's `id()` returns the memory address of an object. When an observer object
is deallocated (garbage collected), Python may allocate a *new* observer at the
same memory address. The cache sees the same `id()` and returns the position
computed for the old (now-dead) observer.

This is fundamentally different from the `set_topo()` cache bug fixed in
v1.0.0a6, which was about the same cache key being reused when `set_topo()`
explicitly replaces the observer. This bug occurs even without `set_topo()` —
any observer object that gets garbage collected can cause a collision.

```python
# Before: cache key = (id(observer), jd_tt)
# id() can be reused after deallocation → stale hit

# After: cache stores (observer, result) and validates identity
if cached_observer is observer:  # identity check, not equality
    return cached_result
```

**Fix:** Cache now stores `(observer, result)` tuples. On lookup, it validates
`cached_observer is observer` (Python identity check). A cache hit where the
object at the same address is a *different* object is treated as a miss.

## Bug Fix 3: Asteroid LEB Comparison Test Date Range

**File:** `tests/test_leb/compare/conftest.py` — `_ASTEROID_SPK_JD_START`

**Symptoms:** 10 asteroid comparison tests failed with `EphemerisRangeError`:
- 5 in `test_extended_asteroids.py` (Chiron, Ceres, Pallas, Juno, Vesta)
- 5 in `test_compare_leb_asteroids.py` (same bodies)

**Root cause:** `_ASTEROID_SPK_JD_START` was set to `year_to_jd(1920)`, but
Horizons SPK21 files for these asteroids begin coverage around 1925. The test
date generation sampled dates in 1922-1923 that fell outside the SPK range.

**Fix:** Changed `_ASTEROID_SPK_JD_START` to `year_to_jd(1930)`, providing a
safe 5-year margin above the actual SPK21 coverage start.

## Bug Fix 4: Fixed Stars Example Tuple Unpacking (Latent)

**File:** `examples/fixed_stars.py`

**Symptoms:** Running `python examples/fixed_stars.py` raised
`unsupported format string passed to tuple.__format__` — the script crashed
before producing output. This was a latent bug (no test covered this exact
code path in the examples test suite until now).

**Root cause:** `swe_fixstar_mag()` returns `(magnitude: float, star_name: str)`,
matching the Swiss Ephemeris API. The example assigned the full tuple to `mag`
and passed it to `f"{mag:.2f}"`, which fails because tuples don't support float
format specifiers.

**Fix:** `mag, _ = eph.swe_fixstar_mag(star_name)` — unpack the tuple.
