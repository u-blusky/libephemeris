## Bug Fix 1: ASC Near-360° Normalization

**Files:** `libephemeris/houses.py` — `swe_houses()`, `swe_houses_armc()`

**Symptoms:** Whole Sign house cusps offset by exactly 30° when the Ascendant
falls at the 0°/360° boundary (e.g., ARMC=270°, lat=60°, eps=23.44°).

**Root cause:** The ASC value `359.99999999999994°` passed the `>= 360.0`
check due to IEEE 754 floating-point representation. `floor(359.999.../30)*30`
evaluates to 330° instead of 0°, shifting all Whole Sign cusps by one sign.

**Fix:** Added epsilon-aware normalization: `if asc >= 360.0 or (360.0 - asc) < 1e-10: asc = 0.0`
Applied to both `swe_houses()` and `swe_houses_armc()`.

## Bug Fix 2: Topocentric house_pos with Body Latitude

**File:** `libephemeris/houses.py` — `house_pos()`

**Symptoms:** When a celestial body has non-zero ecliptic latitude (e.g., the
Moon at ±5°), `house_pos()` for Topocentric (T) diverged from pyswisseph by
up to 0.25 house position units.

**Root cause:** Topocentric fell through to the default cusp-based fallback,
which interpolates using ecliptic longitude only. This ignores the body's
Right Ascension shift caused by non-zero ecliptic latitude.

**Fix:** Added explicit semi-arc handler for Topocentric, using the same
Placidus semi-diurnal/nocturnal arc method. For house *position* purposes
(not cusp computation), Topocentric and Placidus are mathematically equivalent.

---

## Bug Fix 3: Alcabitius house_pos with Body Latitude

**File:** `libephemeris/houses.py` — `house_pos()`

**Symptoms:** `house_pos()` for Alcabitius (B) diverged from pyswisseph by up
to 0.084 house position units with non-zero body ecliptic latitude.

**Root cause:** Same fallback issue as Topocentric — ecliptic longitude
interpolation between cusp longitudes.

**Fix:** Implemented RA-based interpolation: convert cusp ecliptic longitudes
to Right Ascension, then interpolate the body's RA between cusp RAs. This is
correct because Alcabitius cusps are equally spaced in RA per quadrant
(trisection of the ASC's diurnal/nocturnal semi-arc in RA).
